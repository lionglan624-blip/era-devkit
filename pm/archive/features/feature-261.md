## Philosophy (変更不可)

### なぜ「全ERB」なのか

> **「口上ERBだけで十分」は許容しない。**

1. **過去の失敗**: 「十分」と判断した範囲の外に問題が存在し、同じ問題が繰り返し発生した
2. **100%の信頼**: 部分的な調査では「調査していない範囲」への不信が残る
3. **完全な区画**: 本 Feature 完了後、「全ERBを調査済み」と断言できる状態を作る

### なぜ「LLM直接確認」なのか

> **Grep/正規表現は完全性を保証できない。**

1. **パターン漏れ**: 想定外の関数命名パターンを見逃す可能性
2. **コメントアウト**: `;@KOJO_*` の扱いが曖昧
3. **エンコーディング**: Shift_JIS 等で誤動作の可能性

**唯一の完全な方法**: LLM が全ファイルを直接 Read し、人間の目で確認するのと同等の精度で調査する。

### なぜ「意味的監査」なのか

> **ファイル配置だけでは不十分。口上内容と COMF 定義の整合性が必要。**

1. **SOURCE/EXP 整合**: COM の SOURCE:快Ｖ/快Ａ/快Ｂ と口上描写が一致しているか
2. **重複検出**: 同一関数が複数ファイルに存在しないか
3. **内容品質**: 口上がそのCOMの行為を正しく描写しているか
4. **主客関係**: COMF の PLAYER/TARGET 定義と口上の行為者/被行為者が一致しているか → **F269 で実施**

### 教訓 (2025-12-30 追記)

> **検証項目自体の網羅性を担保するプロセスがなかった。**

F261 設計時、項目 1-3 のみ定義し項目 4（主客関係）が漏れた。
F265 実装中に K6 COM 43 で主客逆転が発覚し、F269 として追加調査が必要になった。

「完全な調査」を謳うなら、検証項目自体の完全性も事前に検証すべきだった。

cat .tmp/f261-p1-v3/*.txt | grep -c '^@KOJO_MESSAGE_COM'      # → 1475

2. 各分割を順次 Read
3. 全分割の内容を統合して監査
4. 1 つの JSONL 行として出力

##### Output Format (JSONL)

各関数につき 1 行の JSON を出力:

```json
{"file":"1_美鈴_KOJO_K1_愛撫.ERB","func":"@KOJO_MESSAGE_COM_K1_60","com":60,"chara":"K1","A":"OK","C":"OK","D":"OK","issue":null}
{"file":"2_小悪魔_KOJO_K2_口挿入.ERB","func":"@KOJO_MESSAGE_COM_K2_60","com":60,"chara":"K2","A":"NG","C":"OK","D":"OK","issue":"A: 正規配置は _挿入.ERB"}
```

| フィールド | 型 | 説明 |
|------------|-----|------|
| file | string | 元 ERB ファイル名 |
| func | string | 関数名 |
| com | number | COM 番号 |
| chara | string | キャラ識別子 (K1-K10, KU) |
| A | string | 配置チェック結果 (OK/NG/SKIP) |
| C | string | SOURCE 整合結果 (OK/NG/SKIP) |
| D | string | 品質チェック結果 (OK/NG/SKIP) |
| issue | string/null | 問題詳細 (なければ null) |

**B 重複チェックは Phase ③ で実施** (全関数の file+func を集約して判定)

cat .tmp/f261-p1-v3/*.txt | grep -o 'KOJO_MESSAGE_COM_K[0-9U]*_[0-9]*' | \

; 1. メイン関数を呼び出し (COM 番号に対応)

IF RESULT

### F267 CANCELLED: Wrapper パターン誤解 (2025-12-31)

**現象**: F267 は「K4 NTR 口上がスタブ状態」として作成されたが、実際には実装済みだった。

**原因**: Phase ② 監査が wrapper 関数と implementation 関数を個別に評価した結果の誤解釈。

| 関数 | D 判定 | 実態 |
|------|:------:|------|
| `@NTR_KOJO_MESSAGE_COM_K4_302` | NG | Wrapper (CALL TRAIN_MESSAGE + CALL _1) |
| `@NTR_KOJO_MESSAGE_COM_K4_302_1` | OK | 実装本体 (50+ lines NTR dialogue) |

**ERA Wrapper パターン**:
```erb
@NTR_KOJO_MESSAGE_COM_K4_302      ; wrapper - PRINTFORM なし (正常)
    CALL TRAIN_MESSAGE
    CALL NTR_KOJO_MESSAGE_COM_K4_302_1
    ...

@NTR_KOJO_MESSAGE_COM_K4_302_1    ; 実装本体 - 口上あり
    PRINTFORML ...
```

Phase ② は両方を正しく評価したが、F267 作成時に wrapper NG のみを抽出し「スタブ」と誤解。

**教訓**: `_1` suffix 関数は wrapper の実装本体。wrapper の D: NG は「PRINTFORM がない」だけで「実装不足」ではない。Wrapper + implementation ペアで評価すべき。
