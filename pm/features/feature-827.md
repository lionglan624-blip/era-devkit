## Background

### Philosophy (Mid-term Vision)

Pipeline Continuity is the SSOT for phase transition: each completed phase triggers planning for the next phase, with analysis methodology adapted to the phase type. For research/analysis phases (no code migration), planning produces document-creation sub-features and obligation triage rather than CALL/TRYCALL dependency graphs.

### Problem (Current Issue)

The current F827 DRAFT was auto-generated by F814 using the F783/F814 migration planning template (`pm/features/feature-827.md:31,48`), which assumes CALL/TRYCALL/CALLFORM analysis to derive inter-file dependencies for ERB-to-C# code migration. However, Phase 23 is a research/analysis phase ("NTR Kojo Reference Analysis") with no ERB files to migrate (`docs/architecture/migration/phase-20-27-game-systems.md:385-389`). Its deliverables are two reference documents (`pm/reference/ntr-kojo-analysis.md` and `pm/reference/ntr-ddd-input.md`), not C# code. The CALL/TRYCALL analysis methodology is meaningless because Phase 23 analyzes NTR kojo branching patterns (IF/ELSEIF structures within files), not inter-file call chains. Additionally, all 11 Mandatory Handoffs from F826 are routed to F827 (`feature-826.md:613-623`), but most are code migration stubs (IVariableStore 2D, INtrQuery, NullMultipleBirthService, etc.) belonging to Phase 24-27, not Phase 23 research scope.

### Goal (What to Achieve)

Decompose Phase 23 into research sub-features that produce NTR kojo branch analysis documents, Phase 24 DDD design input, and content-roadmap 8h/8m/8n gap analysis updates, plus transition features (Post-Phase Review Phase 23 + Phase 24 Planning). Analysis methodology uses NTR branch pattern analysis (FAV/TALENT/situation condition counting in ERB kojo files) instead of CALL/TRYCALL dependency graphs. (Note: F826's deferred obligations are handled by F829 per C6, not F827.)
