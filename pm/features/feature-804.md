## Background

### Philosophy (Mid-term Vision)

Phase 21 Counter System C# migration is the SSOT for all counter subsystem logic. F804 (WC Counter Core) migrates the toilet counter's action selection, validation, pose, punishment, reaction, and event kojo dispatch to C#, establishing the WC-parallel architecture alongside F801's Main Counter Core. All WC Counter Core logic must be fully expressed in Era.Core with equivalence-verified C# implementations.

### Problem (Current Issue)

The F804 DRAFT was auto-generated by F783's Phase 21 decomposition with planning-phase line count estimates rather than empirical file measurements, resulting in 5 of 7 Scope Reference line counts being significantly wrong (TOILET_COUNTER_ACTABLE.ERB is 973 lines, not 609; TOILET_COUNTER_COMBINATION.ERB is 11 lines, not 212). More critically, the WC Counter subsystem has independently evolved with 2.9x more validation complexity than the Main Counter (ACTABLE:19-22 repeat-check pattern appears 30 times, ACTABLE:177-202 contains 13 character-specific location checks), and requires 5 interface methods not present in ICounterUtilities.cs (TimeProgress, RestGetUrge, GetDateTime, GetTargetNum, MasterPose) plus CSTR, EXPLV, and NOITEM accessors not in any current interface, because F783 optimized for decomposition breadth over implementation-readiness analysis, and F801's scope covered only the simpler Main Counter utility set.

| Source File | Actual Lines | DRAFT Lines | Description |
|-------------|:------------:|:-----------:|-------------|
| TOILET_COUNTER.ERB | 381 | 360 | Main WC counter flow |
| TOILET_COUNTER_ACTABLE.ERB | 973 | 609 | WC action validation (2.9x Main ACTABLE) |
| TOILET_COUNTER_COMBINATION.ERB | 11 | 212 | WC combination stub ("つくりかけ") |
| TOILET_COUNTER_POSE.ERB | 131 | 168 | WC pose selection |
| TOILET_COUNTER_PUNISHMENT.ERB | 45 | 88 | WC punishment handling |
| TOILET_COUNTER_REACTION.ERB | 131 | 235 | WC reaction processing |
| TOILET_EVENT_KOJO.ERB | 69 | 69 | WC counter event dialogue |
| Total | 1741 | 1741 | Individual counts wrong; total coincidentally correct |

### Goal (What to Achieve)

Migrate all 7 WC Counter Core ERB files to C# classes in Era.Core/Counter/ following F801's established architecture pattern (DI-based, interface-segregated), extending ICounterUtilities with 5 additional methods (TimeProgress, RestGetUrge, GetDateTime, GetTargetNum, MasterPose) and IVariableStore with 3 new accessors (GetCharacterString, GetExpLv, GetNoItem), and producing equivalence tests that verify behavioral parity with the legacy ERB implementation.

### Constraint Details

**C1: Shared CounterActionId Enum**
- **Source**: CounterActionId.cs defines all CNT_ constants; WC ACTABLE uses same IDs
- **Verification**: Grep TOILET_COUNTER_ACTABLE.ERB for CNT_ values and confirm all map to existing enum
- **AC Impact**: No new enum entries; ACs should verify WC uses existing CounterActionId values

**C2: Shared ICounterUtilities (Additive Extension)**
- **Source**: ICounterUtilities.cs currently has 5 methods; WC needs 5 additional methods (TimeProgress, RestGetUrge, GetDateTime, GetTargetNum, MasterPose)
- **Verification**: Count methods before and after F804
- **AC Impact**: AC must verify backward compatibility (existing 5 methods unchanged) and new methods added

**C3: WC Repeat-Check Pattern**
- **Source**: TOILET_COUNTER_ACTABLE.ERB:19-22 -- WC_前々回セクハラ/WC_前回セクハラ CFLAGs checked 30 times
- **Verification**: Grep for WC_前々回 and WC_前回 in ACTABLE
- **AC Impact**: Equivalence tests must verify repeat-action blocking logic

**C4: COMBINATION Stub**
- **Source**: TOILET_COUNTER_COMBINATION.ERB is 11 lines, marked "つくりかけ" (work-in-progress)
- **Verification**: Read TOILET_COUNTER_COMBINATION.ERB -- only loop scaffold
- **AC Impact**: AC verifies stub class exists with minimal logic; no complex behavior expected

**C7: F805 MESSAGE Stub Interface**
- **Source**: TOILET_EVENT_KOJO.ERB:33,41,61,63 calls EVENT_WC_COUNTER_MESSAGE (F805 scope)
- **Verification**: Grep TOILET_EVENT_KOJO.ERB for EVENT_WC_COUNTER_MESSAGE
- **AC Impact**: IWcCounterOutputHandler interface must be defined; F805 provides implementation

**C13: CSTR Accessor Gap**
- **Source**: TOILET_COUNTER_ACTABLE.ERB:352 uses CSTR (character string variable)
- **Verification**: Grep IVariableStore.cs for CSTR or GetCharacterString -- not found
- **AC Impact**: IVariableStore extension needed; AC must verify method exists

**C14: EXPLV Accessor Gap**
- **Source**: TOILET_COUNTER_ACTABLE.ERB:695 uses EXPLV (experience level)
- **Verification**: Grep IVariableStore.cs for EXPLV or GetExpLv -- not found
- **AC Impact**: IVariableStore extension needed; EXPLV is a 1D global lookup table (not per-character), so the method signature is `Result<int> GetExpLv(int level)` -- AC#29 enforces the 1D int parameter by matching `GetExpLv\(int`

- **Source**: TOILET_COUNTER_ACTABLE.ERB:223 (`ITEM:0 == 0 && ITEM:2 == 0 && ITEM:10 == 0`), :477 (`ITEM:PBAND != 0 || ITEM:4 != 0`), :485 (same)
- **Verification**: IItemVariables.cs already has `GetItem(int itemId)` (F776)
- **AC Impact**: WcActionValidator constructor must include IItemVariables parameter; AC must verify injection

## Technical Design

### Approach

F804 follows F801's established DI-based, interface-segregated architecture pattern exactly. Each of the 7 ERB source files maps to one C# class in `Era.Core/Counter/`. The implementation proceeds in three layers:

**Layer 1 - Interface Extensions (no new implementations, only declarations):**
Extend `ICounterUtilities` with 5 new method signatures (TimeProgress, RestGetUrge, GetDateTime, GetTargetNum, MasterPose), extend `IVariableStore` with `GetCharacterString` (CSTR) and `GetExpLv` (EXPLV), and add a `GetNoItem` accessor. Define `IWcCounterOutputHandler` stub for F805's MESSAGE calls. All extensions are additive-only; no existing signatures are touched.

**Layer 2 - Core Logic Classes:**
`WcActionSelector` migrates `TOILET_COUNTER.ERB` (381 lines). The WC main flow differs from Main Counter in several ways: it has WC-specific COMABLE categories (soft/hard/clothed vs. Main's soft/clothed/undressingCaress), a separate "HARD" category (CNT_お尻いじめ group, lines 20-29), love component (愛情), and excitement component (興奮) used in weight formulas. Sex weight is always forced to 0 (line 333). The WcActionSelector calls `WC_ACTABLE` via `WcActionValidator`, dispatches to `WcCounterPose` via direct injection (matching ERB's CALL structure), and dispatches to `WcCounterPunishment/WcCounterReaction` via `IWcCounterOutputHandler` (matching ERB's EVENT dispatch structure).

`WcActionValidator` migrates `TOILET_COUNTER_ACTABLE.ERB` (973 lines). This is the largest and most complex class. It extends the F801 ActionValidator pattern with: (1) WC repeat-check guards using `CFLAG:WC_前々回セクハラ/WC_前回セクハラ` checked 30 times across cases, (2) `TIME_PROGRESS` calls for time-gated actions, (3) 13 character-specific location checks for `CNT_パンツ選択`, (4) `NOITEM`, `CSTR`, `EXPLV` accesses, and (5) `DATETIME()` guard. The WC ACTABLE has additional action IDs (CNT_囁く, CNT_お尻いじめ, CNT_パンツを脱がす, CNT_パンツ選択, CNT_アイテム装着される, CNT_スカートたくし上げ命令, CNT_貞操帯関連, CNT_飲尿命令) that do not exist in the Main ACTABLE.

`WcCounterCombination` migrates `TOILET_COUNTER_COMBINATION.ERB` (11 lines) as-is: a stub FOR loop that accumulates action type counts from active TARGET characters. No complex logic; the ERB is marked "つくりかけ".

`WcCounterPose` migrates `TOILET_COUNTER_POSE.ERB` (131 lines): TFLAG increment by action ID, plus 69-position transition logic via a private `HandlePose69` helper.

`WcCounterPunishment` migrates `TOILET_COUNTER_PUNISHMENT.ERB` (45 lines): random-slot dispatch to 3 named variants (WC_PUNISHMENT_0/1/2) using a while-loop pattern equivalent to ERB TRYCALLFORM. The 3 named functions (panty removal, vibrator insertion, anal vibrator insertion) are implemented as private methods in the same class.

`WcCounterReaction` migrates `TOILET_COUNTER_REACTION.ERB` (131 lines): 5 pose-conditional branches (vaginal, masturbation, fellatio, paizuri, frotteurism), each calling `ICounterUtilities.MasterPose` to determine active pose.

`WcEventKojo` migrates `TOILET_EVENT_KOJO.ERB` (69 lines): `KOJO_MESSAGE_WC_COUNTER` function that sets co-spectator CFLAG, calls descriptive style helpers (stubbed), and dispatches to `IWcCounterOutputHandler.SendMessage`. The action-29 branch (乳搾り手コキ) passes an argument of 1.

**Layer 3 - Equivalence Tests:**
Follow the F801 pattern from `ActionSelectorTests.cs` and `ActionValidatorTests.cs`. Each class gets a corresponding `Wc*Tests.cs` file in `Era.Core.Tests/Counter/`. Test stubs (`WcSelectorTestVariableStore`, `WcSelectorTestEngineVariables`, etc.) follow the same naming convention as F801's `SelectorTestVariableStore`. Key test cases target: repeat-check blocking (C3), COMABLE phase weights (C10), sex weight=0 (C11), TRYCALLFORM dispatch (C12), character-specific location checks (C6), and MasterPose branching (C20).

This approach satisfies all 58 ACs: Layer 1 satisfies AC#9, AC#11-17, AC#28-31, AC#39 (13 ACs); Layer 2 satisfies AC#1-8, AC#10, AC#18, AC#20-25, AC#32, AC#36, AC#37, AC#40, AC#42-49, AC#55, AC#56, AC#57, AC#58 (32 ACs); Layer 3 satisfies AC#19, AC#27, AC#33-35, AC#38, AC#41, AC#50-54 (12 ACs); AC#26 is satisfied by zero-warning builds enforced by TreatWarningsAsErrors (1 AC). Total: 58 unique ACs.
